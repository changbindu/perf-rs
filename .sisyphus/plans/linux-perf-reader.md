## Objective

Implement a Linux perf.data file reader that can parse files generated by the Linux perf tool and our own perf-rs writer, enabling the `report` and `script` commands to work with standard perf.data files.

---

## Background

The perf-data-compatibility plan successfully implemented a **writer** that generates Linux perf compatible files. However, the `report` and `script` commands currently cannot read these files because there's no Linux perf format **reader**.

Current state:
- ✅ Writer: Generates valid Linux perf.data files
- ✅ Validation: Files can be read by `perf report` and `perf script`
- ❌ Reader: Not implemented (causing the error)

---

## Architecture

### Key Components

1. **PerfDataReader**: Main reader struct for parsing perf.data files
2. **Header Parser**: Reads and validates file header
3. **Attribute Parser**: Parses perf_event_attr structures
4. **Event Parsers**: Parses SAMPLE, MMAP, COMM, FINISHED_ROUND events
5. **Symbol Resolver**: Resolves symbols from ELF binaries (already exists)
6. **Integration**: Wire up to `report` and `script` commands

### Data Flow

```
perf.data file
    ↓
PerfDataReader::from_path()
    ↓
Parse header → Parse attrs → Parse events
    ↓
Build event stream (SAMPLE, MMAP, COMM, etc.)
    ↓
Symbol resolution
    ↓
Report/Script output
```

## Technical Details

### File Format Constants

```rust
const PERF_FILE_MAGIC: u64 = 0x50455246494c4532; // "PERFILE2"
const PERF_FILE_HEADER_SIZE: u64 = 104;
const PERF_ATTR_SIZE_VER8: u64 = 136;
const PERF_EVENT_HEADER_SIZE: u16 = 24; // type(2) + misc(2) + size(2) + padding(18)
```

### Event Types to Support

Priority order (implement in this order):
1. **PERF_RECORD_SAMPLE** (9) - Core sampling data
2. **PERF_RECORD_MMAP** (1) - Memory mappings for symbol resolution
3. **PERF_RECORD_COMM** (3) - Process command names
4. **PERF_RECORD_FINISHED_ROUND** (68) - Batch boundaries
5. Future: LOST, EXIT, FORK, etc. (as needed)

### Sample Fields (bit-ordered)

```rust
bitflags! {
    pub struct SampleType: u64 {
        const IP = 1 << 0;
        const TID = 1 << 1;
        const TIME = 1 << 2;
        const ADDR = 1 << 3;
        const READ = 1 << 4;
        const CALLCHAIN = 1 << 5;
        const ID = 1 << 6;
        const CPU = 1 << 7;
        const PERIOD = 1 << 8;
        const STREAM_ID = 1 << 9;
        const RAW = 1 << 10;
    }
}
```

### Alignment Rules

- All records aligned to 8 bytes
- Strings null-terminated and padded
- Use `byteorder` crate for explicit little-endian

## Edge Cases

1. **Empty perf.data files**: Handle gracefully (no samples)
2. **Large files**: Use streaming parser, don't load entire file into memory
3. **Missing symbols**: Display raw addresses when symbols unavailable
4. **Corrupted data**: Validate magic number, detect malformed records
5. **Kernel version differences**: Handle variable attr sizes (64-144 bytes)
6. **Feature bits**: May need to skip unknown feature sections

---

## Performance Considerations

1. **Memory**: Use streaming parser, not DOM-style loading
2. **Speed**: Memory-mapped I/O for large files (optional)
3. **Lazy parsing**: Only parse events when needed
4. **Symbol cache**: Cache resolved symbols to avoid repeated lookups

---

## Backward Compatibility

Must maintain support for old format (PERFRS01):
- Auto-detect format based on magic number
- Use separate parser for old format
- Provide deprecation warning for old format

```rust
enum PerfDataFormat {
    LinuxPerf,     // PERFILE2
    PerfRsLegacy,  // PERFRS01 (deprecated)
}
```
